**시간 절약 명목의 우회로 제안 금지** + **"이정도면 충분하지~" 수준의 타협 금지**. 항상 정공법으로 한 번에, 완벽하게 구현.

**Why:**
- 사용자가 여러 차례 (HWPX 통합 양식, rhwp 셀 wrap 디버깅 등) "시간 절약 이딴거 고려하지 말라", "타협하지 말고 정공법으로", "이정도면 충분하지~ 가 아니라 완벽하게 구현해" 를 명시.
- 시간/규모/난이도를 이유로 우회로 추천 → 사용자가 원치 않는 미봉책 방향으로 드리프트.
- 부분 해결한 뒤 "여기까지면 됐다" 식 보고 → 사용자가 발견 못 한 결함이 누적되어 결국 다시 처음부터 손대야 함.

**How to apply:**
- "데모 안정성", "MVP 1단계", "추후 리팩토링" 같은 우회 표현 금지.
- 어렵거나 깊은 모듈 (예: rhwp 수식 엔진, 탭 정렬 처리) 도 회피하지 말고 들어가서 끝까지 고친다. "이건 별도 작업이라 이번엔 패스" 식 변명 금지.
- 부분 해결 시 미해결 항목을 명확히 나열하고 우선순위를 묻되, 사용자가 "충분"이라 명시하기 전까지는 완성으로 보고하지 않는다.
- 작업 단계 자체는 todo 로 쪼갤 수 있지만 제품 동작·시각 결과는 한 번에 완성된 형태로.
- 막히면 정직하게 "이 부분 막혔다" 보고 — 우회로 추천 X.

**땜질 금지 — 구조적 해결 우선 (2026-05-07 강조)**:
- 증상 위치에 패치 덧대기 금지. 원인이 발생한 **구조적 위치** (parser/AST/layout 단계 중 어디인지) 에서 고친다.
- 특수 케이스 분기 (`if 이 조건이면 다르게 처리`) 로 회피 금지. 일반화된 규칙을 찾아 모델 자체를 바꾼다.
- 매직 상수 튜닝 (gap을 0.3 → 0.85 처럼 눈으로 맞추기) 만으로 끝내지 말고, 그 상수의 출처/의미를 코드 또는 원본 spec 으로 검증한다.
- "수정 범위가 크다 / 다른 곳을 건드린다" 는 회피 사유가 안 됨 — 사용자가 "꽤 건드리는 건 상관없다" 명시.
- 제안 단계에서도 "이게 땜질인지 구조 수정인지" 스스로 검증한 뒤 보고. 본질적이지 않은 제안은 처음부터 하지 말 것.
