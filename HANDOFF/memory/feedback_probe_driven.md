구현 전에 가설을 **probe (픽셀 측정 / 레이아웃 dump / 단위 테스트 / 원본 spec 확인 등) 로 검증** 하고, 검증된 사실에만 기반해 코드 작성. "아마 이렇겠지" 로 큰 변경을 시작하지 말 것.

**Why:**
- 사용자가 명시 (2026-05-07): "차근차근 probe를 통해 검증해 나가면서 정확히 밝혀진 사실 기반으로만 구현해."
- 이전 세션에서 cases gap 24px 차이 원인을 추정만으로 "Tab 폭 줄이면 된다" 고 진단했다가 측정해보니 실제 원인은 eqalign 우측 정렬 + 셀 height inflation 이었음 — 측정 없이 fix 시작했으면 잘못된 방향으로 갈 뻔했음.
- 한컴 같은 외부 시스템의 의미론을 추측으로 결정하면 wrong-by-default — 원본 출력의 픽셀이 ground truth.

**How to apply:**
- 가설 세우기 → probe 작성·실행 → 결과로 가설 입증/반박 → 입증된 만큼만 구현. 반박되면 가설 수정 후 재 probe.
- probe 종류: pdf 픽셀 측정 (PIL/numpy), 레이아웃 dump (debug test 작성), AST 출력, 원본 PDF 와 user 출력 같은 좌표에서 직접 비교.
- "이 정도 측정했으니 이게 맞을 것" 식 일반화 금지 — 한 케이스 측정으로 전체 모델 결정 X. 여러 행/여러 문항으로 cross-check.
- 구현 중에도 중간 산출물(레이아웃 dump 등) 으로 의도한 변화가 일어났는지 probe — 예측과 다르면 즉시 멈추고 원인 확인.
- probe 결과는 보고에 정확한 수치로 인용 (paren center y=781,879,963 / gap=98,84 식). 추상적 "거의 비슷함" 표현 금지.
- 사실로 확정되지 않은 부분은 "추정" / "가설" 임을 명시하고 검증 전엔 코드 변경하지 않음.
